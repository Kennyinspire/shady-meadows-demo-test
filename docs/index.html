Task 1

Verification approach & user story alignment

How I would confirm scenarios are correct and aligned with user needs

1.1 — Understand the user stories before writing a single test

Before touching any tooling, I would hold a short refinement session (or asynchronous review) with the product owner and developers to walk through each user story and its acceptance criteria. The goal is to surface ambiguity early. Specific questions I would raise for this feature:

User Story 1 — questions

The date inputs appear to be free-text (DD/MM/YYYY) — should they also accept YYYY-MM-DD or only the displayed format? Is the "Check Availability" button disabled until both dates are filled? What is the minimum/maximum date range the user can select? Does the calendar prevent past dates? What constitutes a "valid" date range (e.g. minimum 1 night)?

User Story 2 — questions

When availability results are returned, do they replace the "Our Rooms" section or appear above/below it? How should the UI communicate "no rooms available" — a message, an empty list, or a banner? Should partially-booked rooms (e.g. one night free in a 3-night range) be shown or hidden? Is the availability driven purely by the API or also by front-end filtering?

User Story 3 — questions

The "Admin" link in the top nav suggests the login page is accessible via /admin or similar. Is login rate-limited? Does a failed attempt show a generic "invalid credentials" message (secure) or specific field errors (potential username enumeration risk)? The 5-second SLA — is this measured server-side or including full page render?

1.2 — Map acceptance criteria to testable conditions (BDD style)

I convert each acceptance criterion into a Given/When/Then statement. This creates a shared language between product, dev, and QA, and later maps directly to automated test cases.

User Story 1 — AC 1 (widget visible)

Given a user navigates to automationintesting.online on a desktop browser
When the page loads
Then they can see the "Check Availability & Book Your Stay" section without scrolling, with two date inputs pre-filled with today's date and tomorrow's date

User Story 1 — AC 3 (valid dates → results)

Given a user is on the landing page
When they enter a valid check-in and check-out date, and click "Check Availability"
Then a list of available rooms is displayed within 3 seconds, each showing a room name, price per night, and a "Book now" button

User Story 2 — AC 3 (no rooms → message)

Given all rooms (Single, Double, Suite) are booked for the selected date range
When the user clicks "Check Availability"
Then the page displays a clear message informing the user that no rooms are available for those dates — and no room cards are shown

User Story 3 — AC 1 (auth within 5s)

Given a user is on the admin login page (/admin)
When they enter valid admin credentials and click Login
Then they are authenticated and the admin panel is visible within 5 seconds of clicking the button

1.3 — Exploratory test charters

A
Charter: Date input validation & format tolerance

The inputs appear to be free-text. Explore what happens with invalid formats (e.g. 2026-06-28, June 28, 28-06-2026), non-date strings, single-digit days/months without leading zeros, and copy-pasted dates from a calendar app. Does the system silently ignore the input, show an error, or crash?

B
Charter: "No availability" state discovery

Pre-seed bookings for all three room types (Single, Double, Suite) for a future date range, then use the Availability Checker to search that range. Document exactly what the UI shows — is there a specific message, an empty list, or nothing at all? This informs TC-04.

C
Charter: Concurrent availability & race conditions

Open two browser sessions simultaneously. Book the last available room in session A and observe what session B displays — is availability updated in real time or on next search?

D
Charter: Admin login error messaging

Submit invalid credentials and inspect the error message. It should be generic ("Invalid credentials") and not reveal whether the username or password was wrong (username enumeration risk). Also test: SQL injection in both fields, very long strings (>500 chars), and Unicode characters.

1.4 — Definition of Ready & Definition of Done

Definition of Ready (before sprint)
  • Acceptance criteria are unambiguous and testable
  • Exact "no availability" message wording agreed with PO
  • Date input format documented (DD/MM/YYYY confirmed as required)
  • Admin login URL confirmed (/admin)
  • Test environment stable; API Postman collection accessible
  • Edge cases discussed and captured as separate ACs or out-of-scope decisions
Definition of Done (before sign-off)
  • All P1 automated tests passing in CI
  • All ACs have at least one passing automated test
  • Performance threshold validated (auth ≤5 s) via k6
  • 0 critical axe-core violations on landing page
  • Exploratory testing completed with no P1/P2 open defects
  • Playwright HTML report archived as CI artefact
  • Regression suite not degraded
Task 2

Automated functional & non-functional scenarios

Runnable Playwright + Newman + k6 test scripts are included in the repo alongside this document

Execution: Yes — the test scripts in this submission are real, runnable code (not pseudocode). See README.md for setup and run instructions. The suite runs against automationintesting.online by default.

2.1 — Automation pyramid strategy

I structure automation across three levels. More tests at the lower levels keeps the suite fast and reliable; UI/E2E tests cover only the critical user-facing journeys.

E2E / UI Tests (Playwright)~15%
API / Integration — Newman (Postman) ~35%
Contract & health checks — Newman / Pact ~50%

2.2 — Tooling rationale

Playwright (TypeScript) — E2E

Cross-browser (Chromium, Firefox, WebKit), built-in waiting, screenshot/video on failure. Ideal for the date-picker interactions in the Availability Checker which require real browser rendering.

Newman — API tests

The team already has Postman collections for Restful Booker Platform. Newman runs them in CI with zero extra tooling investment. Tests execute in seconds and catch broken API contracts before the slower E2E suite runs.

k6 — Performance

JavaScript-based, integrates naturally with the Node ecosystem, and has first-class GitHub Actions support. Thresholds are defined in code so a breach fails the pipeline automatically — no manual result interpretation needed.

axe-core / @axe-core/playwright — Accessibility

Integrated directly into the Playwright test run. Scans the live rendered DOM (not static HTML), catches WCAG 2.1 AA violations, and reports exactly which elements fail and why.

2.6 — Why some scenarios stay as exploratory (for now)

Deferred from automation
  • Invalid date format behaviour — the system's response to malformed input is unknown. Explore first, then automate once the expected behaviour is agreed.
  • "No availability" message wording — the exact text is not in any AC. The selector cannot be written until the wording is confirmed.
  • Mobile responsiveness — the date inputs on a 375px screen may behave differently. Explore first with Playwright device emulation.
  • Concurrent booking race condition — two users booking the last room simultaneously. Needs a controlled environment and coordination to reproduce reliably.
When they will be automated

Once the behaviour is understood and the expected outcome is agreed in a refinement session, each exploratory finding becomes a scripted test. The typical flow is: explore → raise defect or confirm expected behaviour → write regression test → add to CI.

Mobile E2E tests will be added by extending playwright.config.ts with a Mobile Chrome device project — no new infrastructure needed.

Task 3

CI/CD implementation guide

GitHub Actions pipeline — the same stage structure applies to GitLab CI, Jenkins, or Azure DevOps

This guide uses GitHub Actions as the CI/CD platform (the most common choice for teams using GitHub). The same stage structure applies directly to GitLab CI, Jenkins, or Azure DevOps — only the YAML syntax differs.

3.1 — Pipeline overview

📦
Stage 1 — Install & lint (every PR and push)

Runs first. Installs Node dependencies, compiles TypeScript, runs ESLint. Fails fast — no point running tests against code that does not compile.

🧪
Stage 2 — API tests via Newman (every PR and push)

Runs the Postman collection against the test environment. Executes in seconds, catches broken contracts early. JUnit XML results are uploaded as artefacts. E2E stage only starts if this passes.

🌐
Stage 3 — E2E tests via Playwright (every PR and push)

Runs all .spec.ts files including availability-checker, admin-auth, and accessibility. Chromium only in CI for speed; Firefox/WebKit added on merges to main. Screenshots, videos, and traces captured on failure and uploaded as artefacts (retained 30 days).

⚡
Stage 4 — Performance tests via k6 (nightly only)

Only triggered by the schedule event (nightly cron). Runs the auth load test against the test environment. A threshold breach (p95 > 5 s) fails the pipeline and triggers a Slack notification.

📊
Stage 5 — Notify on failure

JUnit XML results are published to GitHub Actions test summary. Slack notification on failure sent to the QA channel.Only fires on main branch failures to avoid noise from feature branches.

3.2 — GitHub Actions workflow (full .github/workflows/test.yml)

The complete workflow file is included in the repository at .github/workflows/test.yml. The key structure is shown here for reference.
# Triggers: PR to main, push to main/develop, nightly cron on: push: branches: [ main, develop ] pull_request: branches: [ main ] schedule: - cron: '0 2 * * *' # Secrets with safe defaults for public test env env: BASE_URL: ${{ secrets.BASE_URL || 'https://automationintesting.online' }} API_BASE_URL: ${{ secrets.API_BASE_URL || 'https://automationintesting.online' }} ADMIN_USERNAME: ${{ secrets.ADMIN_USERNAME || 'admin' }} ADMIN_PASSWORD: ${{ secrets.ADMIN_PASSWORD || 'password' }} jobs: # Stage 2 — Newman API tests api-tests: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm install -g newman newman-reporter-junitfull - run: | newman run postman/RestfulBookerPlatform.collection.json \ --env-var "baseUrl=${{ env.API_BASE_URL }}" \ --reporters cli,junitfull \ --reporter-junitfull-export results/api-results.xml # Stage 3 — Playwright E2E (only if API tests pass) e2e-tests: needs: api-tests runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: '20' - run: npm ci - run: npx playwright install --with-deps chromium - run: npx playwright test - uses: actions/upload-artifact@v4 if: always() # upload even if tests fail with: name: playwright-report path: playwright-report/ retention-days: 30 # Stage 4 — k6 performance (nightly only) performance-tests: needs: e2e-tests if: github.event_name == 'schedule' runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: grafana/setup-k6-action@v1 - run: | k6 run k6/auth-load-test.js \ --out json=results/k6-results.json env: K6_BASE_URL: ${{ env.API_BASE_URL }} ADMIN_USERNAME: ${{ env.ADMIN_USERNAME }} ADMIN_PASSWORD: ${{ env.ADMIN_PASSWORD }}

3.3 — Repository setup steps (one-time)

1
Create the repository

Create a new public or private GitHub repository. Push the project code from your local to the remote repo. The reviewer can then clone it with git clone and run npm install to get started.

2
Add repository secrets

In the GitHub repo: Settings → Secrets and variables → Actions → New repository secret. Add: BASE_URL, API_BASE_URL, ADMIN_USERNAME, ADMIN_PASSWORD, and optionally SLACK_WEBHOOK_URL. Credentials never appear in the workflow YAML or logs.

3
Verify the first pipeline run

Open a pull request or push to main. Navigate to Actions tab → select the run → confirm all stages pass. The Playwright HTML report will appear as a downloadable artefact.

4
Verify nightly performance run

The k6 stage triggers on schedule only. To test it manually: in Actions, select the workflow and click "Run workflow" — this triggers it immediately without waiting for 02:00 UTC.

3.4 — Environment & test data strategy

Environment tiers
  • Local: Developers run npm run test:api and npm run test:e2e before committing.
  • CI/Test: automationintesting.online (public test environment). API and E2E run here on every PR.
  • Pre-release gate: Full regression + k6 performance run before any release. Manual exploratory sign-off by QE.
Test data management
  • Bookings created via POST /api/booking at test start using the bookings.ts fixture helper.
  • All test bookings use firstname: "AUTOTEST" as a namespace — easy to identify and bulk-delete.
  • Teardown runs in Playwright's afterAll / afterEach hooks via DELETE /api/booking/:id.
  • Credentials stored in GitHub Secrets — never hardcoded in source files.
